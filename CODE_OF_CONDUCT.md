# Code of Conduct

## Purpose

Invaris Quantum ReQISC is a research-software project. Technical disagreement is expected, but contributors must keep review, discussion, and decision-making professional, evidence-focused, and accessible to new contributors.

## Expected behavior

Participants are expected to:

- discuss technical claims with evidence, reproducible examples, or clearly labeled hypotheses;
- distinguish between implemented behavior, proposed behavior, and speculative research direction;
- welcome questions from contributors with different backgrounds in quantum information, compiler engineering, numerical software, documentation, or open-source maintenance;
- give review feedback that identifies concrete issues and actionable changes;
- credit external work, tools, and prior art accurately;
- avoid language that implies institutional affiliation, endorsement, or benchmark performance where no evidence exists;
- respect maintainer decisions about scope, release timing, and research-integrity review.

## Unacceptable behavior

The following behavior is not acceptable in project spaces:

- harassment, intimidation, personal attacks, or sustained disruption;
- insulting contributors based on background, identity, nationality, language, education, employer, seniority, or research field;
- publishing private communications or personal information without permission;
- using project infrastructure for spam, unrelated promotion, or recruitment outside project scope;
- submitting fabricated benchmark results, unverifiable performance claims, or disguised marketing material;
- misrepresenting third-party relationships, sponsorship, endorsements, or access to proprietary systems;
- pressuring maintainers to merge work that lacks tests, citations, or clear scope boundaries.

## Research-integrity expectations

This project treats research-integrity issues as conduct issues when they affect trust in the repository. Examples include fabricated data, copied prose without attribution, exaggerated implementation status, or claims that confuse a scaffold with a validated algorithm.

## Enforcement process

Maintainers may take one or more of the following actions:

1. ask for clarification or edits;
2. move a discussion back to technical scope;
3. hide, edit, or remove comments that violate this code;
4. close or lock issues or pull requests;
5. decline contributions from a participant for a period of time;
6. escalate repeated or severe behavior to repository administrators or hosting-platform support.

Whenever practical, maintainers will explain the reason for enforcement. Severe harassment, threats, private-data exposure, or deliberate falsification may be acted on immediately.

## Reporting

Report conduct concerns by contacting the maintainer address listed in `SECURITY.md` or by opening a private communication channel with a maintainer. Reports should include links, screenshots, dates, and a short description of the concern. Do not use public issues for private safety concerns.

## Scope

This code applies to repository issues, pull requests, discussions, documentation review, project communication channels, and events explicitly organized for the project.
