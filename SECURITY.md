# Security Policy

## Supported versions

This project is currently pre-1.0 research software. Security fixes are applied to the default branch until tagged releases are established.

| Version | Supported |
| --- | --- |
| default branch | Yes |
| older snapshots | No |

## Reporting a vulnerability

Report security concerns privately to: `security@example.org`.

Please include:

- affected version or commit;
- affected component;
- steps to reproduce;
- expected impact;
- whether the issue is already public.

Do not publicly disclose a vulnerability before maintainers have had a reasonable opportunity to respond.

## What counts as a security issue

Security issues may include unsafe file handling, dependency vulnerabilities, malicious benchmark inputs, accidental exposure of secrets in examples, or workflow behavior that could compromise project infrastructure.

Incorrect scientific assumptions, numerical limitations, documentation errors, and benchmark methodology concerns should usually be opened as regular issues unless they create a security impact.

## Response expectations

Maintainers will acknowledge reports when possible, assess impact, prepare a fix or mitigation, and document the resolution in release notes if appropriate.
